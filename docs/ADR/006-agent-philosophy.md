# ADR-006: Agent 设计理念

## 状态

已采纳

## 背景

StyleSwift 需要一个智能体来理解用户意图、调用工具、完成任务。

## 决策

采用 **"模型即智能体"** 理念：

> **The model already knows how to be an agent. Your job is to get out of the way.**

代码只提供能力（工具），模型负责推理决策。

## 原因分析

### 传统方案 vs 模型驱动

| 传统方案 | 模型驱动 |
|---------|---------|
| 预设工作流 | 模型自己决定流程 |
| 代码做决策 | 模型做决策 |
| 硬编码逻辑 | 灵活适应场景 |

### 传统方案的问题

```
预设工作流示例：

1. 用户说"改背景"
2. 代码自动调用 get_page_structure
3. 代码自动调用 apply_styles

问题：
- 用户说"把标题颜色改成红色"，代码仍然先获取整个页面结构？
- 用户说"撤销"，代码如何处理？
- 边界情况越来越多，代码越来越复杂
```

### 模型驱动的优势

```
模型驱动示例：

用户："把标题颜色改成红色"

模型决定：
1. 不需要 get_page_structure（只需要标题信息）
2. 调用 grep(query="h1, h2, h3") 找到标题
3. 调用 apply_styles

用户："撤销"

模型决定：
1. 理解"撤销"意图
2. 调用 apply_styles(mode='rollback_last')

模型根据具体情况推理，而非死板执行预设流程。
```

### 核心设计原则

| 原则 | 说明 |
|------|------|
| 能力原子化 | 每个工具只做一件事，不包含推理 |
| 信任模型 | 不预设工作流，让模型自己决定 |
| 简单输出 | 工具返回事实，不返回判断 |

### 工具设计示例

**好的设计**：
```
get_page_structure → 返回页面结构文本（事实）

模型自己推理：
- 这是导航栏
- 这是内容区
- 需要修改的选择器是 .nav-item
```

**坏的设计**：
```
get_page_structure → 返回 "建议修改 .nav-item 的背景色"（判断）

问题：
- 代码在做推理
- 限制了模型的灵活性
- 可能误判用户意图
```

## 后果

### 优点

- 代码简洁，无复杂工作流
- 灵活适应用户需求
- 模型能力提升时自动获益

### 缺点

- 效果依赖模型能力
- 可能出现意外行为（需要迭代上限保护）
- Token 消耗不可预测

### 技术影响

System Prompt 设计重点：
- 身份定义（你是网页样式个性化智能体）
- 工具说明（每个工具做什么，不做什么）
- CSS 生成指引（使用具体选择器、加 !important）
- 风格技能使用规范

工具设计重点：
- 清晰描述（模型知道工具能做什么）
- 原子性（做一件事）
- 简单输出（返回事实）

## 替代方案

### 方案 A：状态机驱动

预定义状态和转换：
```
空闲 → 分析中 → 生成CSS → 应用中 → 完成
```

放弃原因：
- 无法覆盖所有场景
- 边界情况需要硬编码
- 维护成本高

### 方案 B：混合模式

简单场景用预设流程，复杂场景交给模型。

放弃原因：
- 两套逻辑增加复杂度
- 边界难以定义
- 模型能力足够，无需预设流程
