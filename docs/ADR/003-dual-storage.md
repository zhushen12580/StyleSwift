# ADR-003: 双层存储架构

## 状态

已采纳

## 背景

Chrome 扩展有多种存储选项：
- `chrome.storage.local`：异步 Key-Value 存储，配额约 10MB
- `chrome.storage.sync`：可同步到云端，配额更小（~100KB）
- `IndexedDB`：浏览器内置数据库，配额大（~数百 MB）

## 决策

采用 **双层存储**：

```
┌─────────────────────────────────────┐
│        chrome.storage.local         │
│        （轻量、高频读写）             │
│                                     │
│  • 用户画像、API 设置                │
│  • 会话索引、会话样式                │
│  • 活动会话样式镜像                  │
│  • 风格技能                         │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│            IndexedDB                │
│        （大体积、低频读写）           │
│                                     │
│  • 对话历史 + CSS 快照               │
└─────────────────────────────────────┘
```

## 原因分析

### 数据特征

| 数据 | 大小 | 读写频率 | 特点 |
|------|------|---------|------|
| API Key / 设置 | <1KB | 极低 | 启动时需立即读取 |
| 会话索引 | <5KB/域名 | 中 | 打开面板时展示 |
| 会话样式 | <50KB/会话 | 中 | 恢复会话时注入 |
| 活动样式镜像 | <50KB | 每次刷新 | document_start 阶段读取 |
| **对话历史** | **几十KB~几MB** | 低 | 长期存储 |

### chrome.storage.local 的限制

- 配额约 10MB
- 单个 key 最大 8KB（sync 限制，local 无此限制但整体配额小）
- 适合结构化、小体积数据

### 对话历史的问题

多轮对话后，历史会积累到几百 KB 甚至几 MB：
- 用户画像 + 偏好：几 KB
- 每轮对话：几 KB（含 CSS 快照）
- 50 轮对话 = 几百 KB
- 长期使用 = 几 MB

如果全部存 chrome.storage.local，会快速耗尽配额。

### IndexedDB 的优势

- 配额大（按域名约 50MB+，用户可授权更多）
- 支持结构化存储
- 适合大体积、低频访问数据

## 后果

### 优点

- 存储空间充足，支持长期使用
- 轻量数据读取快速
- 大体积数据不阻塞 UI

### 缺点

- 两套 API 需要分别封装
- 数据一致性需要额外管理

### 技术影响

- IndexedDB 需封装为 Promise 接口
- 会话切换时需分别读写两处存储
- 存储清理需同时处理两处

## 替代方案

### 方案 A：全部用 chrome.storage.local

放弃原因：对话历史会快速耗尽 10MB 配额。

### 方案 B：全部用 IndexedDB

放弃原因：
- 简单数据（如 API Key）读取延迟高
- early-inject.js 在 document_start 阶段读取，IndexedDB 初始化可能未完成

### 方案 C：chrome.storage.sync 同步到云端

放弃原因：
- 配额仅 100KB，严重不足
- 会暴露用户 API Key 到云端
