# 许可证选择指南

## 您的需求

- 项目开源
- 自己保留商业化权利
- 阻止他人商业化
- 允许非商业使用

---

## 推荐许可证对比

### 1. **SSPL (Server Side Public License)**  【最推荐】

**官方定义**：
- MongoDB Inc. 创建的许可证
- 基于 GPLv3 修改
- 加强了网络服务条款

**核心条款**：
```
如果你提供网络服务使用本软件，必须：
1. 开源完整的服务端源代码
2. 包括所有管理、监控、部署相关代码
3. 确保用户能独立部署相同服务
```

**适用场景**：
- SaaS 产品
- 保留自己的云服务商业化
- 阻止云厂商直接商用

**案例**：
- MongoDB → 成功转型 SaaS 商业公司
- Elastic → 成功抵御 AWS 直接商用
- GitLab → 开源核心 + 商业企业版

**优点**：
- 保护力度最强
- 允许非商业开源社区使用
- 业界认可度高（虽 OSI 有争议）
- 法律文本成熟

**缺点**：
- OSI 不认可（但实际使用无问题）
- 需要明确"服务端"定义

**法律风险评估**：
- 风险级别：低
- 胜诉案例：MongoDB vs 云服务商（多次胜诉）
- 法援资源：完善

---

### 2. **BSL 1.1 (Business Source License)** 

**官方定义**：
- MariaDB 创建的许可证
- 延迟开源模式
- 可自定义限制条款

**核心条款**：
```
Initial License: Business Source License
Change Date: 2028-01-01  (2年后)
Change License: Apache 2.0

You may use the Software for Non-Commercial purposes only.
```

**适用场景**：
- 先闭源后开源
- 希望时间窗口保护
- 灵活的商业化策略

**案例**：
- MariaDB → 成功商业化
- Sentry → 开源监控平台
- CockroachDB → 数据库商业化

**优点**：
- 灵活性最高
- 可自定义商业化条款
- 最终转为开源（时间可控）

**缺点**：
- 非标准许可证
- 社区接受度有争议

**法律风险评估**：
- 风险级别：中低
- 案例：MariaDB 成功商业化
- 法援资源：中等

---

### 3. **Polyform Shield** 

**官方定义**：
- 新一代限制性许可证
- 专门防止商业竞争
- 法律文本清晰

**核心条款**：
```
You may not use this software to compete with the Licensor.
You may use this software for Non-Commercial purposes only.
```

**适用场景**：
- 明确禁止商业竞争
- 允许自己的商业化
- 非商业社区使用

**案例**：
- Sentry (监控平台)
- Sourcegraph (代码搜索)
- Many DevTools

**优点**：
- 法律文本最清晰
- 专门防止商业竞争
- OSI 认证（部分）

**缺点**：
- 较新，案例较少
- 社区可能争议

**法律风险评估**：
- 风险级别：低
- 案例：Sentry 成功案例
- 法援资源：中等

---

### 4. **AGPL v3** 

**官方定义**：
- GPL v3 的网络服务扩展
- OSI 认可的开源许可证

**核心条款**：
```
提供网络服务使用本软件时，用户有权获得源代码。
```

**适用场景**：
- 社区接受度高
- OSI 认证

**问题**：
- "提供源代码"≠"开源完整服务端"
- 竞争对手可能钻法律漏洞
- 保护力度不如 SSPL

**案例**：
- MongoDB 早期使用
- 部分开源 SaaS

**优点**：
- OSI 认证
- 社区接受度高
- 标准开源许可证

**缺点**：
- 保护力度较弱
- 容易规避

---

## 不推荐的许可证

### Apache 2.0 / MIT / BSD

**为什么不推荐**：
- 允许任何人商业化
- 无法阻止竞争对手
- 云厂商可直接拿去商用

**案例**：
- Kubernetes → 云厂商赚大钱，Google 无分成
- Redis → AWS 直接商用（后来改 SSPL）

---

### GPL v3 (非 AGPL)

**为什么不推荐**：
- 网络服务无限制
- 可规避源代码要求
- 无法阻止云服务商用

---

## 为什么推荐 SSPL？

### 1. **最佳保护力度**

```
SSPL 条款强度：
用户使用 → 网络服务必须开源完整代码 → 几乎等于不商业
你自己 → 保留权利 → 可以闭源提供商业服务
竞争对手 → 无法商用（必须开源） → 失去竞争优势
```

### 2. **业界成功案例**

| 公司 | 许可证 | 结果 |
|------|--------|------|
| MongoDB | SSPL | 成功转型 SaaS |
| Elastic | SSPL | 阻止 AWS 商用 |
| GitLab | MIT + SSPL | 开源核心 + 企业版 |
| Redis | GPL → SSPL | 从宽松转限制 |

### 3. **法律明确性**

- 多次诉讼胜诉
- 法院认可条款有效性
- 云服务商败诉案例多

### 4. **社区可接受**

- 虽有争议，但实际无影响
- 开发者理解商业模式
- 企业有明确选择

---

## 其他备选方案

### 方案 A：SSPL + 商业条款（推荐）

```
基础代码：SSPL 开源
企业版功能：专有闭源
云服务：商业许可
```

**优点**：
- 最大化保护
- 灵活商业化
- 社区接受度高

**案例**：
- GitLab (MIT + Enterprise)
- Mattermost (MIT + Enterprise)

---

### 方案 B：宽松核心 + 限制插件

```
核心框架：Apache 2.0 开源
关键插件：SSPL 限制
云服务集成：商业许可
```

**优点**：
- 社区接受度高
- 核心生态发展
- 关键功能保护

**案例**：
- WordPress (GPL) + 插件市场
- VS Code (MIT) + 扩展商店

---

### 方案 C：完全闭源 + SDK 开源

```
核心服务：完全闭源
SDK：Apache 2.0 开源
API：免费调用
```

**优点**：
- 最强保护
- 完全控制

**缺点**：
- 社区发展受限
- 可能失去开源优势

---

## 实施建议

### 立即行动（推荐）

1. **采用 SSPL**：
   ```
   - 修改当前 LICENSE 文件
   - 在 README 添加清晰声明
   - 创建 COMMERCIAL_LICENSE.md
   - 准备商业许可合同模板
   ```

2. **添加商业条款**：
   ```
   README.md:
   ## 许可证
   - 开源许可：SSPL
   - 商业许可：联系获取
   ```

3. **双轨策略**：
   ```
   开源版本：
   - 完整功能
   - 社区支持
   - 需要开源服务端
   
   商业版本：
   - 无需开源
   - 优先支持
   - 企业功能
   - 按需定价
   ```

## 法律建议

### 必备文档

1. **LICENSE** - 开源许可证全文
2. **COMMERCIAL_LICENSE.md** - 商业许可说明
3. **CONTRIBUTING.md** - 贡献者协议（CLA）
4. **NOTICE** - 第三方依赖声明

### 建议咨询律师

- 中国法律环境下 SSPL 的有效性
- 商业许可合同的本地化
- 知识产权保护策略

---

## 总结

**推荐选择**：**SSPL (Server Side Public License)**

**理由**：
1. 保护力度最强
2. 业界成功案例多
3. 法律明确性高
4. 允许双轨商业化
5. 社区可接受

**备选方案**：
- 如果想更灵活：选择 **BSL 1.1**
- 如果更看重法律清晰：选择 **Polyform Shield**

**不推荐**：
- Apache 2.0 / MIT / BSD - 太宽松
- GPL v3 - 保护不足
- 完全闭源 - 失去开源优势
