# ADR-005: 会话隔离模型

## 状态

已采纳

## 背景

用户可能在多个网站上使用 StyleSwift，每个网站可能需要多个独立的样式方案。

## 决策

采用 **三级隔离模型**：

```
全局层: 用户画像（跨网站共享）
  │
  └── 域名层: 每个域名的会话列表
        │
        └── 会话层: 单个会话的样式、元数据、对话历史
              │
              └── 活动镜像: 当前活动会话的样式（用于刷新恢复）
```

## 原因分析

### 为什么需要隔离

| 场景 | 需求 |
|------|------|
| 用户在 github.com | 应用深色模式 |
| 用户切换到 stackoverflow.com | 不应该自动应用 github 的样式 |
| 用户在 github.com 有多个方案 | 工作模式、阅读模式分开管理 |

### 为什么不全局共享

```
错误设计：全局单一会话

问题：
- github.com 的样式应用到 stackoverflow.com
- 选择器不匹配，样式无效
- 可能破坏页面显示
```

### 为什么支持多会话

同一网站可能需要不同风格：
- **工作模式**：深色、高对比度、专注
- **阅读模式**：护眼色、大字体、简洁
- **调试模式**：高亮边框、显示区域

用户希望快速切换，而非重新对话。

### 活动镜像的作用

```
场景：用户在 github.com 应用样式后刷新页面

无活动镜像：
  刷新 → 样式丢失 → 需要重新应用

有活动镜像：
  刷新 → early-inject.js 读取 active_styles:github.com → 立即恢复
```

活动镜像存储当前活动会话的样式，与具体会话解耦，支持：
- 刷新后自动恢复
- 新建会话时清空（干净起点）
- 切换会话时同步更新

## 后果

### 优点

- 网站间样式独立，互不干扰
- 支持多会话管理
- 刷新体验无缝

### 缺点

- 存储结构更复杂
- 会话切换需要同步多处状态

### 技术影响

存储 key 设计：
```
全局：
  userProfile                    # 用户画像
  settings                       # API 设置

域名层：
  sessions:{domain}:index        # 会话列表

会话层：
  sessions:{domain}:{id}:meta    # 元数据
  sessions:{domain}:{id}:styles  # 样式
  IndexedDB: {domain}:{id}       # 对话历史

活动镜像：
  active_styles:{domain}         # 当前活动会话样式
```

## 替代方案

### 方案 A：单域名单会话

每个域名只允许一个会话。

放弃原因：
- 无法支持多风格切换
- 用户需要重新对话才能切换风格

### 方案 B：全局会话池

所有会话存在一个池中，不按域名分组。

放弃原因：
- 列表展示混乱
- 不符合用户心智模型（"这是 github 的样式"）

### 方案 C：无会话概念

不持久化样式，每次打开扩展都是新会话。

放弃原因：
- 用户无法保留已打磨的样式
- 刷新后样式丢失
